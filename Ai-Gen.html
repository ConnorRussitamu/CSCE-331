Using AI to Build a Bartender Simulation Game

This page documents how AI coding assistants—GPT, GitHub Copilot, and Gemini—were applied during the development of a first-person bartender game. It highlights where AI accelerated development, where it struggled, and the practices that led to reliable outcomes.

How AI Was Used on This Project

Big takeaway: AI excels at producing focused, testable units—utility methods, state-machine logic, editor tools—while it is not reliable for designing or assembling large, complex systems end‑to‑end.

Core Use Cases

  • Generate single‑responsibility functions (e.g., drink‑mixing validation, price calculations, time-of-day mood modifiers).
  • Refactor & document existing C# scripts for clarity, naming, and XML docs.
  • Edge‑case brainstorming for dialogue branches, customer archetypes, and quest flags.
  • Produce editor tooling (Unity custom inspectors, gizmos, menu items) to speed content iteration.
  • Explain stack traces and suggest likely fixes for null refs or race conditions.

What Stayed Human‑Driven

  • Overall architecture: scene flow, save/load design, event bus, and data schemas.
  • Game feel decisions: pacing, difficulty curves, economy tuning.
  • Content canon: narrative tone, setting, and character voices.
  • Risk management: dependency selection, licensing checks, and security review.

Representative Snippet: Mix Validation

Prompted: “Write a C# function that checks an ordered list of poured ingredients against a recipe allowing ±5ml tolerance and returns a scored match.”

public static (bool isMatch, float score) ScorePour(
    IReadOnlyList<(string ingredient, float ml)> poured,
    IReadOnlyList<(string ingredient, float ml)> recipe,
    float tolerance = 5f)
{
    if (poured.Count != recipe.Count) return (false, 0f);
    float total = 0f, matched = 0f;
    for (int i = 0; i < recipe.Count; i++)
    {
        total += recipe[i].ml;
        if (!string.Equals(poured[i].ingredient, recipe[i].ingredient, StringComparison.OrdinalIgnoreCase))
            continue;
        float delta = MathF.Abs(poured[i].ml - recipe[i].ml);
        float local = MathF.Max(0f, 1f - (delta / MathF.Max(tolerance, 0.01f)));
        matched += MathF.Min(recipe[i].ml, poured[i].ml) * local;
    }
    float score = total > 0 ? matched / total : 0f;
    return (score >= 0.9f, score);
}

This kind of targeted function is where AI shined: fast iteration, easy to test, and low architectural risk.

Strengths & Weaknesses of AI in Coding

Strengths

  • Speed on small scope: accurate boilerplate and utility methods in minutes.
  • Exploratory design: multiple patterns (e.g., ScriptableObject vs. JSON data) quickly compared.
  • Translation & glue code: convert pseudocode to C#, port snippets between APIs.
  • Debug hints: plausible causes for exceptions; quick test scaffolds.
  • Documentation: summaries, comments, and checklists generated on demand.

Weaknesses

  • Global architecture: tends to overfit to the prompt and miss cross‑cutting concerns.
  • Long‑context drift: loses track in multi‑file designs; produces inconsistent types.
  • Fabrication risk: invents APIs or Unity methods that don’t exist.
  • Security & licensing: suggests questionable packages; requires human vetting.
  • Performance naïveté: GC thrash, allocations per frame, or LINQ in hot paths.
Rule of thumb: Let AI draft isolated, testable pieces. Keep system boundaries, data models, and performance budgets as human‑owned design artifacts.

Model Comparison: GPT vs. GitHub Copilot vs. Gemini

Model Where It Shined Where It Struggled Best For Notes
GPT (ChatGPT)
  • Detailed explanations and architectural trade‑offs.
  • Generating complete functions with reasoning and tests.
  • Refactoring suggestions with rationale.
  • Large multi‑file projects without explicit context windows.
  • May produce authoritative‑sounding but incorrect Unity API usage.
 Design discussions, utilities, documentation, unit tests. Use structured prompts and paste only the relevant files or interfaces.
GitHub Copilot
  • Inline speed: autocompletes within the editor in real time.
  • Great at boilerplate, repetitive patterns, and private API usage it sees onsite.
  • Can mirror local smells or propagate anti‑patterns.
  • Weak at global design or cross‑file consistency without curated context.
 Everyday scaffolding, adapters, trivial data models, getters/setters, editor scripts. Pair with tests and static analysis to catch subtle regressions.
Gemini
  • Concise code suggestions and API lookups when given precise prompts.
  • Useful for content ideation (customer archetypes, dialogue themes).
  • May under‑specify types or produce simplistic Unity patterns.
  • Similar long‑context limitations on multi‑file refactors.
 Brainstorming content, short utility code, cross‑checking GPT output. Validate Unity API calls; prefer small, verifiable tasks.

When AI Worked Best

  • Tasks < 60 lines with a single responsibility and clear I/O.
  • Refactoring with acceptance criteria and unit tests provided up front.
  • Generating data assets (JSON/ScriptableObjects) from a schema.

When AI Fell Short

  • Designing the entire save system or event pipeline.
  • Coherently editing 5–10 related files without a map of dependencies.
  • Performance‑critical hot loops or physics/network code.

Practical Guidelines

  1. Define boundaries first. Write the interface, invariants, and test cases; then ask AI to fill the middle.
  2. Keep prompts local. Paste only the code the assistant must see; summarize the rest in bullets.
  3. Constrain outputs. Specify file names, method signatures, and performance budgets (“no GC allocs in Update”).
  4. Prefer many small asks over one big ask. Chain from green tests to broader refactors.
  5. Cross‑verify models. Use Gemini to sanity‑check GPT; use Copilot for inline alternatives.
  6. Track provenance. Keep a log of which files were AI‑assisted for licensing and review.
  7. Never merge blind. Run unit tests, play‑mode tests, and static analysis before accepting AI code.
Summary: AI is a force multiplier for well‑specified, bite‑sized programming tasks. For large systems, human architects should design modules and interfaces, then delegate narrow functions to AI for rapid iteration.
